# Redis 缓存结构规划

依据要求，为推荐系统保障极速响应与冷启动回退策略，设计三种核心缓存，以满足不同层次的数据拉取与写入需求。

## 缓存结构设计表

| 缓存类型 | Key 格式 | 数据结构 | 存储逻辑与说明 | 过期策略 (TTL) |
|---|---|---|---|---|
| **热门排行榜<br>(Cold-Start Fallback)** | `rec:popular:topk` | **Sorted Set** (ZSET) | **逻辑**: 基于线下定时任务计算得出的全站热门榜单 (运用 威尔逊区间算法结合时间指数衰减 生成)。<br>**结构**: `Member` 为 `movie_id`<br>`Score` 为算法评定的热门度得分，从高到低排序，使用 `ZREVRANGE` 提取。 | **TTL: 1小时**<br>通过系统级的后台定时任务 (Scheduled Task) 每小时批量更新 (重新计算热门分数进行覆盖)。 |
| **用户短期会话特征<br>(Session Profiling)** | `user:session:{user_id}:genres` | **Hash** (HASH) | **逻辑**: 记录用户近期交互的核心流派偏好，支持基于实时点击行为做动态特征更新。<br>**结构**: `Field` 为流派标识 (如 `Action`, `Sci-Fi`)<br>`Value` 为该流派的累积交互权重分 (可用 `HINCRBYFLOAT` 动态累加)。 | **TTL: 24小时**<br>自最后一次互动后 24 小时过期，释放不活跃用户的内存开销。若有交互则利用 `EXPIRE` 续期。 |
| **语义缓存<br>(Semantic Cache)** | `cache:semantic:{user_id}` | **String** (STRING, JSON) | **逻辑**: 用户打开主页时请求后端生成的“成形推断列表”。阻挡首页下拉或短时重复刷新对侧边车模型推理的压力。<br>**结构**: Value 格式为 JSON Array 数组模型，例如 `[105, 204, ...]`。 | **TTL: 5分钟**<br>极短效过期机制。保证推荐的新鲜度，又能在极端突发的高并发刷新中提供 O(1) 阻断拦截。 |

---
**架构补充说明**：
- **一致性策略**：所有 Key 必须使用带业务前缀的 Namespace（如 `rec:`, `user:`, `cache:`）进行路由与管理。
- **内存回收**：建议 Redis 配备 `volatile-lru` 策略（仅对设定了过期时间的 key 执行 LRU 淘汰），以防止内存跑满时挤兑非过期核心结构。
